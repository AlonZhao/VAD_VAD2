> **你的人设**：有量产**决策规划**经验的工程师，从规划视角审视端到端，不是从零学 AI 的小白。
>

## 1. 术语对照表：端到端 ↔ 规则规划

把 VAD 的概念逐个翻译成你的母语。**这张表是你的核心武器**，背熟。

| VAD（端到端） | 规则决策规划 | 本质关系 |
|---|---|---|
| BEV 特征 | 栅格地图 / 占据图 / 环境模型 | 都是自车中心的统一环境表示 |
| Detection + Motion Head | 感知 + 预测模块输出的 obstacle list | 障碍物及其预测轨迹 |
| Map Head（向量化车道线） | 高精地图 / 车道拓扑 | 可行驶区域与参考线 |
| ego query | 规划器的"自车状态种子" | 规划的起点表征 |
| ego↔agent cross-attention | 交互式预测 / 博弈 / 风险评估 | "哪些车影响我" |
| ego↔map cross-attention | 参考线选择 / 车道决策 | "我该沿哪条道走" |
| 轨迹回归 (v1) | 轨迹优化输出一条轨迹 | 直接出解 |
| 轨迹词表分类 (v2) | **lattice / 采样候选 + 打分** | 撒点选优 |
| 碰撞约束 loss | 碰撞检测 / 安全距离硬约束 | 别撞 |
| 边界约束 loss | 道路边界 / 可行驶区域约束 | 别出界 |
| 方向约束 loss | 航向对齐 / 参考线贴合 | 别斜着开 |
| min-of-N / 多模态 | 多策略候选（左绕/右绕/跟车） | 多解择优 |
| 匈牙利匹配 | 障碍物 ID 关联 / 数据关联 | 预测和真值配对 |
| L2 误差 | 轨迹跟踪误差 | 跟人类/真值的偏差 |
| 碰撞率 | 安全性指标 | 同义 |

**用法示例**（面试现场）：
> 面试官："VAD 怎么做规划的？"
> 你："本质和我们规则规划一个思路——先有环境模型（BEV，相当于我们的占据图），有障碍物预测（Motion Head，相当于我们的预测模块），有车道（Map Head，相当于高精地图）。然后 ego 去和这些交互——ego↔agent 就是我们做的交互式风险评估，ego↔map 就是参考线/车道决策。只不过它用 attention 隐式学，我们用规则显式算。"

## 2. VADv2 轨迹词表 = 你最熟悉的 lattice 规划 ⭐

这是**整个端到端里和你背景最对口的点**，必须讲到极致。

### 2.1 结构上几乎一模一样

```
规则 lattice 规划：
  ① 在 Frenet 坐标系撒一批候选轨迹（横向偏移 × 纵向速度 网格）
  ② 每条用 cost 函数打分（碰撞 + 舒适 + 效率 + 贴参考线...）
  ③ 选 cost 最低的

VADv2 轨迹词表：
  ① 预定义 4096 条候选轨迹（对训练集聚类得到）
  ② 每条用神经网络打分（含碰撞/边界/中心线/专家模仿多个维度）
  ③ 选分最高的
```

**同构**：都是「**预定义候选 + 打分选优**」，都不是凭空生成轨迹，而是在候选集里选。

### 2.2 关键差异（这里见功力）

| 维度 | lattice（规则） | VADv2 词表 | 你的点评 |
|---|---|---|---|
| 候选生成 | 解析式实时撒点（Frenet） | 离线聚类固定 | 词表覆盖依赖数据，长尾可能没有；lattice 保证覆盖 |
| 打分 | 手工 cost 函数 | 神经网络 | NN 能学复杂交互，但不可解释、难调单项权重 |
| 候选数 | 几百条，可动态调 | 4096 固定 | 词表多但僵硬；lattice 可按场景裁剪 |
| 约束 | 硬约束（直接 prune） | 软打分（高分仍可能不安全） | **这是端到端的命门** |
| 可解释 | 每项 cost 可追溯 | 黑盒分数 | 量产安全验证 lattice 完胜 |

### 2.3 融合判断（面试官最想听的）

> "我看 VADv2 第一反应就是——这不就是个**学出来的 lattice**吗？那融合就很自然了：让 VADv2 输出 Top-K 候选轨迹（它的强项是学到了复杂交互和人类驾驶习惯），然后**用我们规则规划的安全层去 prune**（碰撞硬检测、运动学可行性、道路边界硬约束）。神经网络负责'像人、会交互'，规则负责'兜底、不出事'。这比纯端到端的软约束安全得多，也比纯规则更会处理长尾交互。"

这段话同时证明了：① 你懂 lattice ② 你懂 VADv2 ③ 你有融合的工程判断。**三连击。**

## 3. 用规划视角逐个审视 VAD 的设计

### 3.1 ego↔agent 交互 = 交互式预测/博弈

**VAD 做法**：ego query 和所有 agent 做 cross-attention，attention 权重隐式表达"哪辆车重要"。

**规划视角**：这就是我们做的**交互式预测**或**博弈式决策**——评估哪些障碍物对自车意图有影响（cut-in 风险、汇入冲突、跟车目标）。

**你的点评**：
> "我们规则里做交互，要显式定义'冲突区''让行规则''博弈代价'。VAD 用 attention 让网络自己学谁重要——好处是不用手写交互规则，长尾交互能泛化；坏处是学出来的注意力不可解释，没法保证它一定关注到了那个突然 cut-in 的车。量产里我会要求对'高风险 agent'有显式的兜底检查，不能全信 attention。"

### 3.2 三个约束 loss = 软化的规则约束

**VAD 做法**：碰撞/边界/方向三个 hinge loss，安全不罚、危险才罚。

**规划视角**：这完全是我们 cost 函数里的 collision cost / boundary cost / heading cost，只是变成可微的训练信号。

**你的点评（关键）**：
> "这三个约束我一眼就认得，就是我们 cost 函数的可微版本。但有个本质区别：我们规则里碰撞约束是**硬的**——撞了直接 prune 掉这条轨迹，绝不可能输出。VAD 是**软的**——碰撞 loss 只是让模型'倾向'不撞，训练完它仍可能输出一条碰撞轨迹，只是概率低。这在量产里不可接受。所以我的判断是：约束 loss 可以用来引导训练，但部署时必须再加一层硬的碰撞检测，不能依赖软约束保证安全。"

### 3.3 碰撞约束的 x/y 阈值不同 = 你早就懂的车体几何

**VAD 做法**：碰撞 loss 横向阈值 1.5m、纵向 3.0m。

**规划视角**：这就是车是长条形、纵向要留刹车距离的常识。

**你的点评**：
> "这个 1.5/3.0 的设计我很认同，和我们规划里给自车膨胀（inflation）做碰撞检测是一个道理——纵向留更大 buffer 是因为追尾后果重、刹车需要距离。说明 VAD 作者也是懂驾驶的。不过我们规则里膨胀半径会随速度动态调整，VAD 这里是固定阈值，高速场景可能不够。"

### 3.4 多模态 = 多策略候选

**VAD v1 痛点**：单模态回归，左转右转都行时学成平均（开到中间）。

**规划视角**：这正是我们规划里要避免的——决策不能"既不左也不右"，必须明确选一个。

**你的点评**：
> "v1 这个单模态平均的问题，我们规则规划其实早就规避了——我们会显式枚举'左绕''右绕''跟车'几个策略分支，各自规划再择优，绝不会输出一条中间的妥协轨迹。v2 用轨迹词表分类本质就是回到了'多策略候选'这条路上，和我们的思路殊途同归。这也印证了：规划这件事，多模态/多候选是绕不开的。"

### 3.5 ego 恒在原点 = 自车坐标系

**VAD 做法**：ego query 位置永远 (0,0)。

**规划视角**：自车中心坐标系（Frenet 或车体系），所有障碍物、车道都是相对量。

**你的点评**：
> "这个和我们规划完全一致，都在自车坐标系下作业。好处是平移不变，模型/规则不用关心全局位置。"

## 4. 端到端 vs 规则规划：你的结构化判断

面试官一定会问"你怎么看两者"。给一个**有层次、不偏激**的判断。

### 4.1 端到端在规划上做对了什么

1. **信息无损**：感知→预测→规划传特征不传结果，不像我们模块间传 obstacle list 会丢信息（比如"这个框的不确定性"传不过去）
2. **联合优化**：规划的 loss 能反传去优化感知/预测，让上游"为规划服务"，而不是各自优化各自的指标
3. **交互泛化**：复杂多体交互不用手写规则，数据里学
4. **省特征工程**：不用手调几十个 cost 权重

### 4.2 端到端在规划上做漏了什么

1. **安全无硬保证**：软约束，没有"绝不碰撞"的硬边界
2. **不可解释**：出了问题没法定位是哪一步决策错了（我们规则能逐步追溯）
3. **长尾兜底差**：训练集没有的场景（施工、事故、违章车）泛化不可控
4. **难形式化验证**：没法像规则那样做覆盖性测试和形式化证明
5. **冷启动依赖数据**：规则零数据就能跑，端到端要大量标注

### 4.3 你的落地结论（核心观点，反复用）

> "我的判断是：**端到端做主决策、规则做安全兜底**，分层架构。端到端负责它擅长的——复杂交互、拟人驾驶、长尾感知；规则负责它擅长的——硬安全约束、可解释、可验证。具体到 VADv2，它的轨迹词表 + 打分天然适合这个架构：让它出 Top-K 候选，规则层做碰撞硬检测和运动学校验，过滤后再执行。这既用上了端到端的学习能力，又守住了量产的安全底线。"

## 6. 面试高频追问（规划视角版）

### Q: 你做规则的，端到端这套你能上手吗？
> "模型部分我已经源码级读透了，上手成本低。而真正难的规划落地——安全兜底、交互处理、参考线决策、轨迹平滑——这些我有量产经验，是相通的。我缺的是端到端的训练实操，但这反而是最容易补的部分。"

### Q: 端到端要取代规则规划了，你不慌吗？
> "我不认为是取代，是融合。即便最激进的端到端方案，量产上车也必须有规则安全层兜底——这是法规和安全的底线决定的，不是技术选择。所以会规则规划的人不会过时，反而是能把两者接起来的人最稀缺。我做的就是往这个方向走。"

### Q: 用规划的话讲讲 VADv2 比 v1 好在哪？
> "v1 是'轨迹优化直接出一条解'，遇到多解场景会平均化，这是规划大忌。v2 是'lattice 撒候选 + 打分选优'，天然多模态，而且打分维度（碰撞/边界/中心线/专家）可以单独看——这就给规则介入留了接口，我可以在打分后、执行前插一层硬约束筛选。从规划工程角度，v2 的架构对量产友好得多。"

### Q: VAD 的碰撞约束够安全吗？
> "不够，它是软约束。训练让模型倾向不撞，但不保证不撞。量产里碰撞必须是硬约束——我会在端到端输出后加一层独立的碰撞检测（基于膨胀车体的时空相交判断），任何预测碰撞的轨迹直接拒绝，fallback 到保守策略或 AEB。软约束引导 + 硬约束兜底，两层都要。"

### Q: 你觉得 VAD 哪里设计得最好/最不足？
> "最好：用 query 做 ego↔agent/map 交互，把交互式决策做成了端到端可学的，省了大量手写交互规则。最不足：安全是软约束、不可解释，这对量产是硬伤。还有向量化假设场景可结构化，施工区、非标准路口这种结构化失败的场景会很脆弱——而这些恰恰是规则规划现在还在兜底的长尾。"
