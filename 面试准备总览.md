## ✍️ 一、端到端规划：简历经历怎么写

> **你的真实情况**：规则规划背景，自学 VAD，**没有跑过训练/没有实车/没有改过模型**，但**源码级深读+产出了系统文档**。
>
> **核心原则**：不撒谎说"做过项目"，但把"深度代码研究"诚实地包装成有含金量的经历。撒谎一追问细节就穿帮，诚实+深度反而是加分项。

### 1.1 定位：这是「技术调研/自学」，不是「项目经历」

简历里**不要**写在"项目经验"栏，写在 **"技术学习/自我提升"** 或 **"技术调研"** 栏。一旦放进项目经验栏，面试官默认你做过工程落地，会追问数据、调参、上线，你就被动了。

**❌ 不能写的（会穿帮）：**
- "参与 VAD 端到端自动驾驶项目开发"
- "复现 VAD 并在 nuScenes 达到 L2 0.72m"（你没跑过）
- "优化 VAD 模型，提升推理速度 X%"
- "负责 Planning 模块的设计与实现"

**✅ 可以写的（真实且有含金量）：**
- 源码级研读
- 产出技术文档/架构图
- 形成技术判断和选型思考
- 把规则背景和端到端做对比分析

### 1.2 简历写法（三个版本，按力度递增）

**版本A — 保守稳健（推荐没把握时用）**
```

- 结合规则规划背景，对比分析两种范式的优劣，形成"端到端主决策
  + 规则安全兜底"的技术判断
```

**版本B — 进取（突出深度，适合技术岗）**

主动跟进端到端自动驾驶演进，完成 VAD/VADv2 源码级研究并产出系统

### 1.3 把"只读代码"转化为可信能力的 4 个抓手

面试官信不信你"真的学懂了"，看的不是你说"我读了代码"，而是这些**可验证的产出**：

| 抓手 | 怎么用 | 证明了什么 |
|------|--------|-----------|
| **GitHub 文档** | 面试前发链接/现场展示 | 真读了，且能讲清楚 |
| **手画架构图** | 当场画 BEV→4head→planning | 真理解，非背诵 |
| **讲数据流** | 一个 batch 怎么从图像到 loss | 贯通了，非碎片 |
| **代码细节** | x/y碰撞阈值1.5/3.0、词表`*0.+`贴轨迹 | 抠到了实现 |

> **话术**："我没有实际训练资源，但我把这套代码读得很透，沉淀成了文档和注释（出示）。我可以现在画一下它的数据流，或者讲任何一个模块的实现细节。"

### 1.4 一句话定调（被问"你这段经历是什么性质"时）

> "这是我主动的技术学习和调研，不是工程项目。我没有训练资源去复现，但我做到了源码级理解——能讲清每个模块为什么这么设计、数据怎么流动、loss 怎么算。我把它当成从规则规划转端到端的'技术地基'来打。"

**这个定调的好处**：诚实（不会穿帮）+ 主动（学习能力强）+ 有产出（文档可证）+ 有判断（双范式视角）。

## 🔥 二、针对"只读代码、没实践"的尖锐追问应对

> 这是你最大的风险区。面试官一旦发现你没跑过代码，会用这些问题试探深度。**提前准备好诚实又不掉价的回答**，比临场慌乱强一百倍。核心策略：**承认没实践 → 立刻用理解深度补上 → 把球踢回技术讨论**。

### Q追1: "这个代码你自己跑通过吗？训练过吗？"

> "没有，我没有训练资源（需要多卡 + nuScenes 完整数据，个人条件不具备）。我的重点放在**把代码读透**——理解每个模块的设计意图、数据流、loss 计算。比如训练启动我追到了 mmcv 的 Runner 双层循环、forward_train 怎么调 head 算多任务 loss。如果有环境，我有信心快速跑起来并调试，因为我清楚每一步该出什么、形状是多少。"

**要点**：坦诚 → 强调读透 → 给一个能证明深度的细节 → 表达上手信心。

### Q追2: "那你说的 L2 0.72m、4.5 FPS 这些指标是你跑出来的？"

> "不是，这些是**论文报告的数据**，我引用是为了说明 VAD 的量级。我没有复现条件，所以不会把它说成我的成果。我更关注的是这些指标背后的**评测机制**——比如 L2 是预测轨迹和真值的平均欧式距离，碰撞率是把自车建成矩形去和障碍物占据栅格做重叠检测。我也清楚开环指标的局限：L2 低不代表真能开（不看舒适度、障碍物冻结）。"

**要点**：明确区分"论文数据"和"我的成果"（诚信！）→ 转向你真正懂的评测原理 → 亮局限认知（显成熟）。

### Q追3: "你改过这份代码吗？加过什么功能？"

> "我加了大量中文注释帮助理解（出示），但没有改模型逻辑——因为没法训练验证，乱改没意义。不过我想过几个改进方向，比如 VAD v1 的规划是单模态回归，遇到左转右转都行的场景会学成平均值，v2 用轨迹词表分类解决了这个；如果让我改 v1，我会先引入多模态输出 + min-of-N loss。这些我在文档里有写。"

**要点**：诚实没改逻辑 + 给合理理由（没法验证）→ 但展示你有改进思路（说明真懂）。

### Q追4: "读代码和真正做项目差很多，你怎么补这个差距？"

> "我完全认同，工程落地有大量代码看不到的东西——数据清洗、训练不收敛的调试、显存 OOM、多卡同步、指标对不齐、部署量化掉点。这些我确实没经历过。我的策略是：**先把原理和代码吃透，降低上手成本**，真正做项目时把精力集中在工程问题上，而不是还在理解模型在干什么。另外我的规则规划经验里，调试、对齐、上线这些工程能力是相通的。"

**要点**：不辩解，大方承认差距 → 说清差距具体是什么（证明你知道工程难在哪）→ 用规则经验的可迁移性补位。

### Q追5: "你遇到过这个代码里什么坑或难理解的地方？"

> "有几个。比如一开始我以为 Detection Head 直接输出他车轨迹，后来才发现是 **Motion Head 复用了 Detection 的 query 特征**（hs[-1]），不是输出框。再比如 VADv2 的规划输出，我一度困惑模型怎么'生成'坐标，读到 `outputs * 0. + plan_anchors` 才明白它根本不生成、是直接贴词表轨迹，模型只负责打分。还有 ego query 那段全是 `.permute(1,0,2)`，是为了适配 PyTorch Transformer 的 (seq, batch, feat) 格式。"

**要点**：这题是**送分题**——能答出具体的"困惑→搞懂"过程，最能证明你真读了。准备 2-3 个真实的理解卡点。

### Q追6: "如果给你环境，第一步你会做什么验证你的理解？"

> "我会先跑通单 batch 的 forward，打印每个中间张量的形状，对照我文档里推的维度——比如 bev_embed 应该是 (B,10000,256)、检测 query (B,300,...)、map query (B,100,20,...)。形状对得上，说明我对数据流的理解没错。然后跑几个 iter 看 loss 是否下降、各任务 loss 量级是否合理。再用 only_bev 模式验证时序 prev_bev 的逻辑。"

**要点**：给出具体、可操作的验证步骤 → 证明你不是空谈，是真的清楚每步该出什么。

### Q追7: "你没实践，我怎么信你能干活？"

> "理解我的顾虑。我能给您三个可验证的东西：第一，现在就能画出 VAD 完整数据流，讲清任意模块；第二，我的 GitHub 文档和代码注释是公开的，可以看我的理解深度和表达能力；第三，我有 X 年规则规划的实际工程经验，调试、对齐、上线这些硬功夫是有的，只是把领域知识从规则迁移到端到端。我缺的是端到端的训练实操，但这是上手成本最低的部分——原理我已经打透了。"

**要点**：把"信任问题"转成"可验证的证据" → 三点式结构清晰 → 强调真正稀缺的工程能力你有。

## 💼 针对"规则规划转端到端"背景的特殊准备

### 面试官必问的3个问题

**Q1: 你做规则规划的，为什么要转端到端？**

**标准答案框架：**
```
痛点识别：
- 规则规划的局限：手工特征、难以处理长尾、调参困难
- 端到端的优势：数据驱动、联合优化、泛化性更好

个人动机：
- 看到行业趋势（特斯拉FSD、Waymo的端到端尝试）
- 希望掌握新技术栈
- 认为未来是端到端+规则的混合方案

学习路径：
- 系统学习VAD作为代表性工作
- 理解矢量化表示（与规则规划的lattice相似）
- 思考如何融合两种方法
```

---

**Q2: 端到端能解决规则规划的哪些痛点？哪些做不到？**

**标准答案：**

**能解决的痛点：**

| 规则规划痛点 | 端到端方案 |
|------------|----------|
| 手工特征设计 | 自动学习特征 |
| 长尾场景覆盖难 | 数据驱动泛化 |
| 模块误差累积 | 联合优化 |
| 调参工作量大 | 一次训练 |

**做不到的：**

| 需求 | 规则规划优势 | 端到端劣势 |
|------|------------|----------|
| 可解释性 | 每步决策可追溯 | 黑盒模型 |
| 安全验证 | 形式化证明 | 难以验证 |
| 边界情况控制 | 显式约束 | 软约束可能违反 |
| 冷启动 | 无需数据 | 依赖大量标注 |

**结论：** 混合方案最优（端到端提供候选，规则做安全检查）

---

**Q3: VADv2的轨迹词表和规则规划的lattice有什么相似和不同？**

**标准答案：**

**相似之处：**
- 都是预先定义候选轨迹集合
- 都是"选择"而非"生成"
- 都便于加约束筛选

**不同之处：**

| 维度 | Lattice（规则） | Trajectory Vocab（VADv2） |
|------|---------------|------------------------|
| 生成方式 | 解析式（Frenet坐标） | 数据驱动聚类 |
| 评分方式 | Cost函数（手工设计） | 神经网络分类器 |
| 候选数量 | 几百条（实时生成） | 4096条（预先固定） |
| 覆盖范围 | 完全覆盖 | 依赖训练数据 |
| 可控性 | 完全可控 | 部分可控（词表设计） |

**融合思路：**
```python
# VADv2输出Top-K
topk_trajs = VADv2.predict(images, k=10)

# 规则规划筛选
for traj in topk_trajs:
    if rule_based_safety_check(traj):
        return traj

# 如果都不安全，回退到规则生成
return lattice_planner.emergency_trajectory()
```

## 🎤 面试实战技巧

### 回答问题的STAR结构

**S (Situation)**: 问题背景
**T (Task)**: 需要解决什么
**A (Action)**: VAD的方案
**R (Result)**: 效果和局限

**示例：**
```
面试官问："VAD如何解决碰撞风险？"

错误回答：
"VAD用了碰撞约束损失。"（太简短）

正确回答：
【S】传统方法用密集占用图判断碰撞，计算昂贵。
【T】VAD需要高效的碰撞约束方式。
【A】VAD利用矢量化轨迹，直接计算ego与agent的点到点距离，
    横向阈值1.5m，纵向3.0m（符合驾驶习惯）。
【R】训练时作为软约束引导安全轨迹，消融实验显示碰撞率降低29%。
    但仍是软约束，极端情况可能违反，需要规则后处理兜底。
```

---

### 展示思考深度的3个技巧

**1. 对比分析**
```
"VAD的方案是X，相比UniAD的Y更高效，但牺牲了Z..."
```

**2. 批判性思考**

### 常见追问及应对

**追问1: "VAD在真实车上部署有什么挑战？"**

回答要点：
- 计算资源：4.5 FPS还不够，需要优化到30+ FPS

- 安全验证：如何证明端到端模型的安全性？
- 数据闭环：如何持续收集corner case并重训练

- **长期**：结合大模型（如Senna），实现更强的泛化和推理
- **关键挑战**：安全验证、法规合规
