# 技术文档输出方法论

> **用途**：复现"VAD 学习"这次的文档产出模式。换新项目时，复制本文的启动指令即可。
>
> **由来**：学习 VAD 端到端项目时，沉淀出一套"读代码→讲解→落文档"的高质量协作流程，产出了 10+ 篇技术文档。本文把方法固化下来，可迁移到任何项目。

---

## 0. 一句话总纲

> **先读真实代码验证，再下结论；一次只啃一个模块；用我的母领域语言类比；每块都提炼成可用的判断/话术。**

最核心一条：**先读代码再讲，下结论给出 `文件:行号` 证据**。这是文档可信度的根。

---

## 1. 两类文档（先想清楚要哪种）

| | 技术型文档 | 视角型文档 ⭐ |
|---|---|---|
| 例子 | `VAD模型架构详解.md` | `从规划视角解读端到端.md` |
| 内容 | 这个模块怎么工作 | 用我的领域语言重新解读它 |
| 可替代性 | 高（网上一搜一堆） | 低（带个人背景判断，独一份） |
| 价值 | 证明你学了 | 证明你有专业判断力 |

- **技术型**：每个项目都该有，记录"是什么、怎么工作"。
- **视角型**：当你有深厚母领域、且新技术与之实质重叠时做，这是差异化武器。

---

## 2. 通用启动指令（换项目改 3 个 `[]`）

开新项目时，把这段发给 AI：

```
我在学习 [项目名]，背景是 [你的背景]，目标是 [学习目标]。

请按这套方式帮我：
1. 先读真实代码/配置再讲解，不要凭记忆，下结论前给出 文件:行号 证据
2. 一次只讲一个模块，讲完我确认后再写文档、commit & push
3. 讲解时用我熟悉的 [领域] 概念做类比，并指出对应/差异/可融合点
4. 明确区分"代码事实"和"你的推测"
5. 每个模块结尾提炼成 [面试话术/技术判断/落地建议]
6. 复杂逻辑用 ASCII 流程图 + 数据维度标注

先从 [入口文件：配置/main/README] 出发，理清整体结构，
再由我指定深入哪个模块。
```

---

## 3. 视角型文档专用指令（追加在通用指令后）

当项目和你的母领域有实质重叠时，追加：

```
另外帮我单独产出一篇"从[我的领域]视角解读[新技术]"的文档：
1. 先做术语对照表：把[新技术]的核心概念逐个翻译成[我的领域]
   的对应物，并标注本质关系和差异
2. 找出和我背景"最同构"的那个点，深挖（结构对应/关键差异/融合方案）
3. 逐个审视[新技术]的关键设计，每个都给"它这样做→我们领域是XX
   →好处→落地我会怎么补"的带视角点评
4. 输出我的"独特价值主张"：我的[领域]背景如何互补纯[新技术]工程师
5. 结尾给[领域]视角版的面试高频追问+答案
```

---

## 4. 分阶段工作流（你主导节奏）

```
阶段0 摸地形    从入口(配置/main)看整体结构，列出模块清单
阶段1 定主线    你决定先啃哪个(挑核心/最对口的)
阶段2 读+讲     AI读真实代码，用类比+流程图讲，你随时打断追问
阶段3 落文档    讲清后写成独立文档(定位+受众+话术)
阶段4 沉淀      commit & push，更新进度
   ↑ 回到阶段1，下一个模块
阶段5 横向      全部啃完后，做技术对比/演进图谱
阶段6 应用      针对目标(面试/选型)做专项产出
```

**节奏由你控制**：用"继续 / 深入X / 换个模块 / 整理文档"驱动。

---

## 5. 视角型文档的核心配方（3 要素）

复盘 `从规划视角解读端到端.md` 为什么成立：

### 5.1 术语对照表（灵魂）
把新领域每个概念翻译成母语领域的对应物：
```
新概念A ↔ 你熟悉的B  |  本质关系  |  你的点评
```
这张表是地基——让你"张口就能用母语重述对方"。

### 5.2 找"最同构"的锚点深挖
每个跨领域学习都有一个"这不就是我们的XX吗"的爆点（VAD里是"词表=lattice"）。找到它、深挖它（结构对应/关键差异/融合方案），这是最可信的切入口。

### 5.3 逐个设计点下"带视角的判断"
不是"它这样做"，而是：
```
"它这样做 → 我们领域是XX → 好处是… → 但落地我会…"
```
每句都暴露工程经验，这是纯算法工程师给不出的。

---

## 6. 三个让产出更值钱的习惯

1. **追问"为什么这么设计"**：逼出的内容恰是面试最爱考的。多问"为什么"，少问"是什么"。
2. **暴露你的困惑**：你卡住的点 = 别人也会卡的点 = 最该写进文档的点。
3. **结合背景下判断**：每个技术点问一句"这和我做过的XX比怎么样"。带视角的判断才是独一份。

---

## 7. 什么项目值得做视角型文档

满足这两条最值得：
1. 你有一个**深厚的母领域**
2. 新技术和母领域有**实质重叠**（都在解决同一类问题）

否则术语对照表会硬凑，价值打折。

---

## 8. 五个隐性成功要素（自检清单）

产出文档时对照检查：

- [ ] 从真实代码出发（读了再讲，不凭记忆）
- [ ] 增量、单主题（一次一个模块）
- [ ] 翻译成母语（用熟悉领域类比）
- [ ] 诚实标注边界（区分"代码事实"vs"推测"）
- [ ] 目标导向（每块提炼成可用的话术/判断）

---

## 9. 一句话记住

> 技术型文档让你"看起来懂"，视角型文档让你"看起来是这个领域的人"。
> 先读代码保证不出错，用母语类比保证记得住，下判断保证独一份。
