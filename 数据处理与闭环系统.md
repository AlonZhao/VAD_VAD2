## 1. 核心概念：三种"推理场景"

理解数据流的关键，是先区分三种"看似都是模型在跑"的场景：

| 场景 | 目的 | 数据来源 | 是否需要 GT |
|------|------|---------|------------|
| **训练** | 学习模型参数 | nuScenes 标注数据集 | ✅ 全部 GT 都需要 |
| **评测推理** | 在测试集上算指标 | nuScenes test 集 | ✅ 需要 GT 算指标 |
| **实车部署** | 真实车辆决策 | 实时传感器流 | ❌ 完全不需要 GT |

**最常见的误解**：以为论文代码里 `forward_test` 就是"部署代码"。

**真相**：`forward_test` 是**评测代码**，部署需要工程师剥离评测逻辑、补齐时序管理。

## 2. 训练数据流

### 2.1 完整数据流图

```
┌─────────────────────────────────────────────┐
│ 第1层：nuScenes 原始数据                      │
│  - 6个相机的 .jpg 图像（2Hz 关键帧）          │
│  - 3D 检测框人工标注（nuScenes在点云上标注）  │
│    ⚠️ 注意：VAD只读取标注结果，不读取点云     │
│  - HD Map（车道线、边界、人行道）             │
│  - Ego 位姿轨迹（每帧 x, y, θ）               │
│  - CAN 总线数据（速度、加速度、转向角）        │
└────────────────────┬────────────────────────┘
                     ▼
┌─────────────────────────────────────────────┐
│ 第2层：预处理脚本（一次性运行）                │
│  tools/data_converter/vad_nuscenes_         │
│    converter.py                             │
│  ↓ 生成 .pkl 文件                           │
│  nuscenes_infos_temporal_train.pkl          │
└────────────────────┬────────────────────────┘
                     ▼
┌─────────────────────────────────────────────┐
│ 第3层：Dataset 类（训练时每次取数）            │
│  VADCustomNuScenesDataset.__getitem__()     │
│  - 读 pkl 中的一条记录                       │
│  - 加载 6 张图像                             │
│  - 应用 pipeline（增强、归一化、坐标变换）     │
│  - 返回字典                                  │
└────────────────────┬────────────────────────┘
                     ▼
┌─────────────────────────────────────────────┐
│ 第4层：DataLoader 打包                       │
│  - 多个样本拼成 batch                        │
│  - workers 并行加载                          │
└────────────────────┬────────────────────────┘
                     ▼
┌─────────────────────────────────────────────┐
│ 第5层：模型 forward                          │
│  VAD.forward_train()                        │
│   → forward_pts_train()                     │
│   → VADHead 计算预测 + loss                  │
└─────────────────────────────────────────────┘
```

### 2.2 训练数据字段详解

每个字段的含义、来源、形状：

| 字段 | 含义 | 来源 | 形状 |
|------|------|------|------|
| `img` | 6相机多帧图像 | 原始 .jpg 文件 | `[B, T, 6, C, H, W]` |
| `img_metas` | 图像元数据（标定、位姿） | 标注 + 计算 | list of dict |
| `gt_bboxes_3d` | 3D 检测框 | nuScenes 人工标注 | `[B, N_obj, 9]` |
| `gt_labels_3d` | 物体类别 | nuScenes 人工标注 | `[B, N_obj]` |
| `map_gt_bboxes_3d` | 矢量化地图 | HD Map 采样为 polyline | `[B, N_map, 20, 2]` |
| `map_gt_labels_3d` | 地图类别 | HD Map 语义 | `[B, N_map]` |
| `ego_his_trajs` | 自车历史轨迹 | 从 ego pose 计算（过去2秒） | `[B, T_h, 2]` |
| `ego_fut_trajs` | 自车未来轨迹 GT | 从 ego pose 计算（未来3秒） | `[B, T_f, 2]` |
| `ego_fut_masks` | 未来轨迹有效性 | 标记场景边界 | `[B, T_f]` |
| `ego_fut_cmd` | 高层指令 | 从未来轨迹推断 | `[B, 3]` one-hot |
| `ego_lcf_feat` | 自车低频特征 | CAN 总线 + 计算 | `[B, 9]` |
| `gt_attr_labels` | 他车未来轨迹 + 多模态标签 | 离线追踪 + 多模态聚类 | `[B, N_obj, ...]` |

### 2.3 关键洞察：哪些 GT 需要标注，哪些是计算来的？

**这是数据闭环的关键！**

#### 类别 A：免费午餐（可以从原始数据自动计算）

```python
# 不需要任何额外标注
ego_his_trajs    # 从 ego pose 历史切片
ego_fut_trajs    # 从 ego pose 未来切片（事后回看）
ego_fut_cmd      # 从 ego_fut_trajs 的横向位移推断
ego_lcf_feat     # 从 CAN 总线日志读取
```

**关键代码示例（伪代码）：**
```python
def compute_ego_gt(ego_pose_log, t):
    """所有 ego_* 字段都可以这样算出来"""
    return {
        'ego_his_trajs': ego_pose_log[t-4:t],
        'ego_fut_trajs': ego_pose_log[t:t+6],
        'ego_fut_cmd':   infer_command(ego_pose_log[t:t+6]),
        'ego_lcf_feat':  can_bus_log[t],
    }
```

#### 类别 B：需要算法/人工标注

```python
gt_bboxes_3d         # 3D 检测框（nuScenes 人工标注）
                     # ⚠️ 标注在LiDAR点云上，VAD只读取标注结果
gt_labels_3d         # 物体类别
map_gt_bboxes_3d     # 矢量化地图（从 HD Map 采样）
map_gt_labels_3d     # 地图类别
gt_attr_labels       # 他车未来轨迹（需要追踪算法）
```

**重要说明**：
- VAD是**纯视觉方法**，不使用LiDAR点云
- 3D框GT虽然是在点云上标注的，但VAD只读取标注结果
- 详见：`VAD数据使用说明.md`

### 2.4 nuScenes 预处理脚本做了什么？

**入口文件**: `tools/data_converter/vad_nuscenes_converter.py`

**核心工作**：
1. 遍历 nuScenes 所有 sample（关键帧）
2. 对每个 sample 计算所有 `ego_*` 字段（事后回看）
3. 把 HD Map 采样为 20 个点的 polyline
4. 把 3D 检测标注整理成 VAD 需要的格式
5. 调用追踪算法生成多模态 agent 轨迹
6. 序列化为 .pkl 文件

**生成的 .pkl 文件结构**：
```python
{
    'infos': [
        {  # 一条 sample 信息（对应一个关键帧）
            # ============================================================
            # 【基础信息】
            # ============================================================
            'token': str,                    # sample唯一标识
            'prev': str,                     # 上一帧的token（''表示场景开始）
            'next': str,                     # 下一帧的token（''表示场景结束）
            'lidar_path': str,               # LiDAR文件路径（VAD不用，但保留）
            'frame_idx': int,                # 当前帧在场景中的序号
            'scene_token': str,              # 所属场景的唯一标识
            'timestamp': int,                # 时间戳（微秒）
            'map_location': str,             # 地图名（如'boston-seaport'）
            'fut_valid_flag': bool,          # 未来6帧是否都存在

            # ============================================================
            # 【坐标变换矩阵】（BEV编码时用于3D→2D投影）
            # ============================================================
            'lidar2ego_translation': [3],    # LiDAR→ego平移 (x,y,z) 米
            'lidar2ego_rotation': [4],       # LiDAR→ego旋转 (四元数qw,qx,qy,qz)
            'ego2global_translation': [3],   # ego→global平移 (x,y,z) 米
            'ego2global_rotation': [4],      # ego→global旋转 (四元数)

            # ============================================================
            # 【CAN总线数据】[18维向量]
            # ============================================================
            'can_bus': np.array([18]),
            # [0:3]   pos: ego位置 (x,y,z) in global
            # [3:7]   orientation: ego朝向 (四元数 qw,qx,qy,qz)
            # [7:9]   accel: 加速度 (ax, ay)
            # [9:11]  rotation_rate: 角速度 (ωx, ωy)  
            # [11:13] vel: 速度 (vx, vy)
            # [13:15] 其他CAN字段
            # [15:17] 其他CAN字段  
            # [17]    yaw角（最后一位，用于时序对齐）
            # 注：具体字段取决于nuScenes CAN bus消息格式

            # ============================================================
            # 【6个相机信息】
            # ============================================================
            'cams': {
                'CAM_FRONT': {
                    'data_path': str,                   # 图像文件路径
                    'type': str,                        # 相机类型
                    'sample_data_token': str,           # sample_data的token
                    'sensor2ego_rotation': [4],         # 相机→ego旋转(四元数)
                    'sensor2ego_translation': [3],      # 相机→ego平移(米)
                    'sensor2lidar_rotation': np.array([3,3]),  # 相机→LiDAR旋转矩阵
                    'sensor2lidar_translation': np.array([3]), # 相机→LiDAR平移
                    'cam_intrinsic': np.array([3,3]),   # 相机内参矩阵
                    #  [[fx, 0, cx],
                    #   [0, fy, cy],
                    #   [0,  0,  1]]
                },
                'CAM_FRONT_RIGHT': {...},
                'CAM_FRONT_LEFT': {...},
                'CAM_BACK': {...},
                'CAM_BACK_LEFT': {...},
                'CAM_BACK_RIGHT': {...},
            },

            # ============================================================
            # 【LiDAR sweeps】（VAD不用，保留兼容性）
            # ============================================================
            'sweeps': list[dict],            # 历史LiDAR扫描信息

            # ============================================================
            # 【3D检测GT】 - 人工标注
            # ============================================================
            'gt_boxes': np.array([N_obj, 7]),
            # [x, y, z, w, l, h, yaw]
            # x,y,z: 中心坐标（LiDAR坐标系，米）
            # w,l,h: 宽度、长度、高度（米）
            # yaw: 朝向角（弧度）

            'gt_names': np.array([N_obj]),   # 类别名称字符串
            # 如 ['car', 'pedestrian', 'truck', ...]

            'gt_velocity': np.array([N_obj, 2]),  # 速度 [vx, vy]（m/s）
            # 注：与gt_boxes拼接后变成9维送入模型

            'num_lidar_pts': np.array([N_obj]),   # 每个框内的LiDAR点数
            'num_radar_pts': np.array([N_obj]),   # 每个框内的Radar点数
            'valid_flag': np.array([N_obj]),      # 有效标志（点数>0）

            # ============================================================
            # 【他车运动GT】 - 追踪算法生成
            # ============================================================
            'gt_agent_fut_trajs': np.array([N_obj, 12]),  # float32
            # 每个agent未来6步轨迹，每步(Δx,Δy) → 6×2=12维
            # 格式：[Δx₁,Δy₁, Δx₂,Δy₂, ..., Δx₆,Δy₆]
            # offset格式：每步相对上一步的位移

            'gt_agent_fut_masks': np.array([N_obj, 6]),   # float32
            # 未来6步的有效性掩码，0或1
            # 0=该步无效（物体消失/场景结束）

            'gt_agent_lcf_feat': np.array([N_obj, 9]),    # float32
            # agent当前状态特征：
            # [0:2] x, y: 当前位置（LiDAR坐标系）
            # [2]   yaw: 当前朝向
            # [3:5] vx, vy: 速度
            # [5:8] width, length, height: 尺寸
            # [8]   type: 类别ID（car=0, truck=1, ...）

            'gt_agent_fut_yaw': np.array([N_obj, 6]),     # float32
            # 未来6步的朝向变化（Δyaw）

            'gt_agent_fut_goal': np.array([N_obj]),       # float32
            # 终点方向类别：0-7=8个方向(45°一段), 9=静止

            # ============================================================
            # 【自车历史轨迹GT】 - 从ego pose计算（免费午餐⭐）
            # ============================================================
            'gt_ego_his_trajs': np.array([2, 2]),         # float32
            # 过去1秒的2步位移（offset格式）
            # [0]: (Δx, Δy) 从t-2到t-1的位移
            # [1]: (Δx, Δy) 从t-1到t的位移
            # 坐标系：LiDAR坐标系
            # 时间：nuScenes 2Hz，每步0.5秒

            # ============================================================
            # 【自车未来轨迹GT】 - 从ego pose计算（免费午餐⭐）
            # ============================================================
            'gt_ego_fut_trajs': np.array([6, 2]),         # float32
            # 未来3秒的6步位移（offset格式）
            # [i]: (Δx, Δy) 从t+i到t+i+1的位移
            # 坐标系：LiDAR坐标系
            # 时间：nuScenes 2Hz，每步0.5秒

            'gt_ego_fut_masks': np.array([6]),            # float32
            # 未来6步的有效性掩码，0或1
            # 0=该步无效（场景结束）

            # ============================================================
            # 【自车驾驶指令】 - 从未来轨迹推断（免费午餐⭐）
            # ============================================================
            'gt_ego_fut_cmd': np.array([3]),              # float32
            # one-hot编码：[右转, 左转, 直行]
            # [1,0,0]=右转（累积横向位移>2m）
            # [0,1,0]=左转（累积横向位移<-2m）
            # [0,0,1]=直行（累积横向位移在±2m内）

            # ============================================================
            # 【自车低频特征】 - CAN+pose计算（免费午餐⭐）
            # ============================================================
            'gt_ego_lcf_feat': np.array([9]),             # float32
            # [0] vx:     横向速度 (m/s) - 从pose微分
            # [1] vy:     纵向速度 (m/s) - 从pose微分
            # [2] ax:     横向加速度 (m/s²) - 从CAN
            # [3] ay:     纵向加速度 (m/s²) - 从CAN
            # [4] w:      角速度 (rad/s) - 从pose微分
            # [5] length: 车辆长度 (m) = 4.084（固定值）
            # [6] width:  车辆宽度 (m) = 1.85（固定值）
            # [7] v0:     纵向速度 (m/s) - 从CAN直接读取
            # [8] Kappa:  转向曲率 (1/m) - 从CAN方向盘角度计算
        },
        ...  # 所有 sample
    ],
    'metadata': {...}
}
```

## 2.5 训练数据的完整分类与溯源

### 2.5.1 训练所需数据全貌

`forward_pts_train()` 接收的所有数据可以分为**5大类**：

```python
losses_pts = self.forward_pts_train(
    # 类别1: 已提取的图像特征（模型内部生成）
    img_feats,
    
    # 类别2: 3D检测相关GT（人工标注）
    gt_bboxes_3d,           # 3D检测框
    gt_labels_3d,           # 物体类别
    gt_bboxes_ignore,       # 忽略的框（难例）
    
    # 类别3: 地图相关GT（HD Map转换）
    map_gt_bboxes_3d,       # 矢量化地图元素
    map_gt_labels_3d,       # 地图元素类别
    map_gt_bboxes_ignore,   # 忽略的地图元素
    
    # 类别4: 自车相关数据（位姿+CAN自动计算）
    ego_his_trajs,          # 自车历史轨迹
    ego_fut_trajs,          # 自车未来轨迹GT
    ego_fut_masks,          # 未来轨迹有效性掩码
    ego_fut_cmd,            # 高层驾驶指令
    ego_lcf_feat,           # 自车低频特征
    
    # 类别5: 他车运动相关GT（追踪算法生成）
    gt_attr_labels,         # 包含他车未来轨迹等
    
    # 类别6: 元数据（标定+位姿）
    img_metas,              # 相机参数、ego位姿等
    prev_bev,               # 上一帧BEV特征（模型缓存）
)
```

### 2.5.2 五大类数据的详细溯源

#### 类别1: 图像特征（模型计算）

| 字段 | 来源 | 生成位置 | 最终形状 |
|------|------|---------|---------|
| `img_feats` | 当前帧图像 → ResNet → FPN | `VAD.extract_feat()` | `list[Tensor]` 多尺度 |
| `prev_bev` | 历史帧图像 → BEV Encoder | `VAD.obtain_history_bev()` | `[B, H*W, C]` |

**数据流**：
```python
img [B, 6, C, H, W]
  → ResNet50 backbone
  → FPN neck
  → img_feats [list of multi-scale features]
```

---

#### 类别2: 3D检测GT（人工标注）

| 字段 | 含义 | 来源 | 最终形状 | 标注成本 |
|------|------|------|---------|---------|
| `gt_bboxes_3d` | 3D检测框 | nuScenes人工标注 | `[B, N_obj, 9]` | 高（$100/场景） |
| `gt_labels_3d` | 物体类别 | nuScenes人工标注 | `[B, N_obj]` | 高 |
| `gt_bboxes_ignore` | 难例/遮挡框 | nuScenes标注 | `[B, N_ignore, 9]` | 高 |

**源头追溯**：
```
nuScenes 原始标注
    ↓ JSON格式
    ↓ 包含：center(x,y,z), size(w,l,h), yaw, 类别
    ↓
预处理脚本: vad_nuscenes_converter.py (line 322-347)
    ↓ 坐标系转换：global → ego vehicle → lidar
    ↓ 格式转换：nuScenes Box → SECOND format
    ↓ 添加速度：计算前后帧位移
    ↓
存入.pkl: info['gt_boxes'], info['gt_names']
    ↓
Dataset加载: nuscenes_vad_dataset.py (line 1230-1238)
    ↓ 过滤无效框（点云数<阈值）
    ↓ 类别名 → 类别ID映射
    ↓
送入模型: gt_bboxes_3d, gt_labels_3d
```

**3D框的9个维度**：
```python
gt_bboxes_3d shape: [N_obj, 9]
# [x, y, z, w, l, h, yaw, vx, vy]
# x,y,z: 中心坐标（ego坐标系）
# w,l,h: 宽度、长度、高度
# yaw: 朝向角
# vx,vy: 速度（2D）
```

---

#### 类别3: 地图GT（HD Map转换）

| 字段 | 含义 | 来源 | 最终形状 | 生成方式 |
|------|------|------|---------|---------|
| `map_gt_bboxes_3d` | 矢量化地图 | HD Map采样 | `[B, N_map, 20, 2]` | 算法自动 |
| `map_gt_labels_3d` | 地图类别 | HD Map语义 | `[B, N_map]` | 算法自动 |
| `map_gt_bboxes_ignore` | 忽略元素 | 规则过滤 | `[B, N_ignore, 20, 2]` | 算法自动 |

**源头追溯**：
```
nuScenes HD Map（矢量格式）
    ↓ 包含：lane, lane_divider, road_boundary, ped_crossing
    ↓
预处理脚本: vad_nuscenes_converter.py (未在提供的代码中)
    ↓ 查询当前ego附近的地图元素（范围如60m×30m）
    ↓ 对每个元素（如车道线）采样固定数量的点（20个）
    ↓ 坐标变换：global → ego vehicle
    ↓
存入.pkl: info['map_anns']
    ↓
Dataset加载: 通过pipeline处理
    ↓ 归一化到BEV范围
    ↓
送入模型: map_gt_bboxes_3d, map_gt_labels_3d
```

## 3. 评测推理数据流

### 3.1 评测推理 vs 训练的区别

```python
# 训练
loss = forward_train(img, gt_bboxes_3d, ego_fut_trajs, ...)
loss.backward()

# 评测推理
predictions = forward_test(img, gt_bboxes_3d, ego_fut_trajs, ...)
metrics = compute_metrics(predictions, gt_bboxes_3d, ego_fut_trajs)
```

**两个关键差异**：
1. **不算 loss**，只算预测 + 评测指标
2. **跨帧时序**靠 `self.prev_frame_info` 维护（而非一次塞 queue）

### 3.2 为什么 forward_test 还传 GT？

**这是初学者最容易困惑的地方。**

代码：`projects/mmdet3d_plugin/VAD/VAD.py` 第 331 行
```python
def forward_test(
    self,
    img_metas,
    gt_bboxes_3d,        # ← 还有！
    gt_labels_3d,
    img=None,
    ego_his_trajs=None,
    ego_fut_trajs=None,  # ← 还有！
    ego_fut_cmd=None,
    ego_lcf_feat=None,
    gt_attr_labels=None,
    ...
):
```

**原因**：传入 GT 是为了**算评测指标**（L2、碰撞率），不是模型输入。

```python
def simple_test_pts(...):
    # 模型预测（不用 GT）
    outs = self.pts_bbox_head(img_feats, ...)
    
    # 这里才用 GT 算指标
    metric_dict = compute_metrics(
        pred_traj=outs['ego_fut_preds'],
        gt_traj=ego_fut_trajs,           # 用来算 L2 误差
        gt_boxes=gt_bboxes_3d,           # 用来算碰撞率
    )
    
    return outs, metric_dict
```

### 3.3 评测时的时序状态管理

**与训练的关键区别**：训练时一次塞入多帧；评测时跨调用维护状态。

```python
class VAD:
    self.prev_frame_info = {
        'prev_bev': None,       # 上一帧 BEV 特征
        'scene_token': None,    # 当前场景 ID
        'prev_pos': 0,          # 上一帧 ego 位置
        'prev_angle': 0,        # 上一帧 ego 朝向
    }
    
    def forward_test(self, ...):
        # 场景切换：清空 prev_bev
        if img_metas[0][0]['scene_token'] != self.prev_frame_info['scene_token']:
            self.prev_frame_info['prev_bev'] = None
        
        # 计算 ego 位姿的相对 delta
        if self.prev_frame_info['prev_bev'] is not None:
            img_metas[0][0]['can_bus'][:3] -= self.prev_frame_info['prev_pos']
            img_metas[0][0]['can_bus'][-1] -= self.prev_frame_info['prev_angle']
        
        # 推理
        new_prev_bev, results = self.simple_test(...)
        
        # 更新状态供下一帧使用
        self.prev_frame_info['prev_bev'] = new_prev_bev
        self.prev_frame_info['prev_pos'] = current_pos
        self.prev_frame_info['prev_angle'] = current_angle
```

```
└────────────────┬────────────────────────────┘
                 ▼
            VAD 模型推理
                 ▼
       ego_fut_preds (未来 3 秒轨迹)
                 ▼
         下游控制模块（横纵向控制器）
```

### 4.3 部署侧 inference wrapper 实现

```python
class VADRealtimeInferencer:
    """实车部署的精简 inference 包装器"""
    
    def __init__(self, model_path):
        self.model = load_vad_model(model_path)
        self.model.eval()
        
        # 时序缓存
        self.prev_bev = None
        self.scene_token = None
        self.prev_pos = np.zeros(3)
        self.prev_angle = 0.0
        
        # 历史轨迹缓冲区
        self.ego_history = EgoHistoryBuffer(history_len=4)
    
    @torch.no_grad()
    def infer(self, sensor_snapshot):
        """每 50ms 调用一次（20Hz）"""
        
        # 1. 解包传感器输入
        imgs = sensor_snapshot['cameras']        # 6 相机
        ego_pose = sensor_snapshot['pose']       # GNSS+SLAM
        ego_speed = sensor_snapshot['speed']     # CAN
        ego_acc = sensor_snapshot['accel']       # CAN
        nav_cmd = sensor_snapshot['nav_command'] # 导航
        
        # 2. 维护历史
        self.ego_history.update(ego_pose)
        
        # 3. 组装模型输入
        img_metas = self._build_img_metas(ego_pose)
        ego_his_trajs = self.ego_history.get()
        ego_lcf_feat = self._build_lcf_feat(ego_speed, ego_acc)
        
        # 4. 时序状态管理（参考 forward_test）
        if self._scene_changed():
            self.prev_bev = None
        if self.prev_bev is not None:
            img_metas['can_bus'][:3] -= self.prev_pos
            img_metas['can_bus'][-1] -= self.prev_angle
        
        # 5. 模型推理（注意：不传任何 gt_*）
        outputs = self.model.simple_test(
            img=imgs,
            img_metas=[img_metas],
            ego_his_trajs=ego_his_trajs,
            ego_fut_cmd=nav_cmd,
            ego_lcf_feat=ego_lcf_feat,
            prev_bev=self.prev_bev,
        )
        
        # 6. 更新缓存
        self.prev_bev = outputs['new_prev_bev']
        self.prev_pos = ego_pose[:3]
        self.prev_angle = ego_pose[-1]
        
        # 7. 返回规划轨迹
        return outputs['ego_fut_preds']
```

研究代码 → 部署代码，工程师需要做：

## 5. 数据闭环系统

### 5.1 什么是数据闭环？

> **数据闭环 = 让"实车遇到的真实数据"自动回流到训练管道，形成"采集→标注→训练→部署→再采集"的循环**

它是**训练数据准备的"持续运转版"**——不再是一次性采集 nuScenes，而是车辆每天都在采集新数据并改进模型。

### 5.2 完整闭环图

### 5.3 模块详解

#### 模块 1：触发器（Trigger）

**作用**：决定什么数据值得回传（避免带宽和成本爆炸）。

**常见触发条件**：

| 触发类型 | 判断逻辑 | 应用场景 |
|---------|---------|---------|
| Disengagement | 人类接管方向盘/刹车 | 模型决策错误 |
| AEB 触发 | 紧急制动启动 | 模型没预判到危险 |
| 模型置信度低 | softmax max < 0.5 | OOD 场景 |
| 预测-实际偏差 | 预测轨迹 vs 实际驾驶轨迹 L2 > 1m | 跟人类不一致 |
| 约束 loss 突增 | 三大约束 loss 大于阈值 | 危险情境 |
| VLM/规则告警 | 规则系统检测到异常 | 施工、临时路障 |
| 随机采样 | 1/1000 概率 | 保证常规场景覆盖 |

**伪代码示例**：
```python
class VADRealtimeInferencer:
    def infer(self, data):
        outputs = self.model(data)
        
        # 数据闭环触发器
        if self._should_upload(data, outputs):
            self.upload_buffer.add({
                'timestamp': time.time(),
                'sensor_data': data,
                'model_output': outputs,
                'trigger_reason': self._diagnose(data, outputs),
            })
        
        return outputs['ego_fut_preds']
    
    def _should_upload(self, data, outputs):
        if self.driver_takeover_detected():
            return True
        if outputs['planning_confidence'] < 0.5:
            return True
        if self.collision_loss(outputs) > THRESHOLD:
            return True
        return False
```

#### 模块 2：自动标注（Auto-labeling）

**这是数据闭环的核心难点和价值。**

车端只能采集到原始传感器数据，没有 GT。要重新进入训练，必须把训练所需的 GT 补出来。

**分两类**：

**类别 A：可以"事后计算"的 GT（免费午餐）**
```python
# 直接从已有数据计算，不需要标注

# ego_fut_trajs: 自车未来3秒轨迹
# → 从车辆 GNSS/SLAM 位姿历史读取（事后回看）
ego_fut_trajs = ego_pose_log[t : t+6]

# ego_his_trajs: 自车历史轨迹
ego_his_trajs = ego_pose_log[t-4 : t]

# ego_fut_cmd: 高层指令
ego_fut_cmd = infer_command_from_trajectory(ego_fut_trajs)

# ego_lcf_feat: 速度/加速度
ego_lcf_feat = can_bus_log[t]
```

**类别 B：需要算法生成的 GT（核心成本）**
```python
# gt_bboxes_3d: 3D 检测框
# → 用大模型/Offline 模型标注（比训练用的小模型强很多）
gt_bboxes_3d = OfflineDetector.predict(
    point_cloud,         # LiDAR
    multi_view_images,   # 6相机
    multi_frame=True,    # 多帧融合
)

# map_gt_bboxes_3d: 矢量化地图
# → 离线建图算法（多次经过同一地点融合）
map_gt = OfflineMapping.build(
    multi_pass_data,
    use_lidar=True,
    use_multi_view=True,
)

# gt_attr_labels: 他车未来轨迹
# → 离线追踪：先前向追踪，再"看"未来轨迹
agent_tracks = OfflineTracker.track(detection_log)
gt_agent_fut_trajs = agent_tracks[t : t+6]
```

**核心原理**：**用大模型标小模型的数据**——这是 Tesla、Waymo、华为的核心壁垒。

#### 模块 3：数据挖掘（Data Mining）

**问题**：车端回传的数据可能仍然 90% 是重复场景。

**解决方案**：再次筛选

```python
class DataMining:
    def mine(self, raw_clips):
        # 1. 去重：相似场景只保留一个
        unique_clips = self.dedup_by_embedding(raw_clips)
        
        # 2. 长尾挖掘：找训练集中稀有的场景
        rare_clips = self.find_rare_scenarios(unique_clips)
        # 例：雨夜 + 施工 + 行人横穿
        
        # 3. 困难样本挖掘：模型预测错的案例
        hard_clips = self.find_hard_cases(unique_clips)
        # 例：当前模型预测 vs 自动标注 GT 偏差大的
        
        # 4. 主动学习：模型置信度低的
        uncertain_clips = self.active_learning_select(unique_clips)
        
        return rare_clips + hard_clips + uncertain_clips
```

#### 模块 4：仿真闭环验证

**问题**：新模型不能直接 OTA 推到车上，否则可能酿成事故。

**解决方案**：在仿真中验证

```python
# 仿真闭环（这也是 RAD 工作的方向）
for scenario in test_scenarios:
    # 用 3DGS 重建真实场景
    sim_env = GaussianSplatting.reconstruct(scenario.sensor_data)
    
    # 让新模型在仿真中开车
    rollout = sim_env.rollout(new_model, steps=200)
    
    # 评测
    if rollout.has_collision or rollout.too_slow:
        reject_model()
```

### 5.4 数据闭环 vs 训练数据准备

回到核心问题：数据闭环和你前面学的"训练数据准备"是什么关系？

| 维度 | 训练数据准备 | 数据闭环 |
|------|------------|---------|
| **数据来源** | 一次性人工采集（nuScenes） | 持续从实车采集 |
| **GT 生成** | 一次性人工标注 | 自动标注 + 部分人工质检 |
| **触发逻辑** | 无（全量入库） | 智能触发（只回传值得的） |
| **闭环性** | 单向（标注→训练） | 双向（部署→采集→训练→部署） |
| **数据量级** | 1k 场景（nuScenes） | 100M+ 场景（特斯拉） |
| **核心目标** | 训练 baseline 模型 | 持续提升模型上限 |

**关系**：
- 训练数据准备 = 数据闭环的**第一次循环**
- 数据闭环 = 训练数据准备的**持续运转版**
- 两者**复用同一套数据格式**（即 `forward_pts_train` 的输入）

### 5.5 数据闭环和 VAD 代码的具体连接

把你看到的代码字段直接对应到数据闭环模块：


## 8. 数据闭环 vs RAD 的关系

VAD 的后续工作 **RAD（NeurIPS 2025）** 把数据闭环推到了下一阶段：

```
传统数据闭环（VAD时代）：
  采集真实数据 → 自动标注 → 模仿学习 → 部署
   ↓ 局限：只能学"人类怎么开"，有上限

RAD 的进化：
  采集真实数据 → 3DGS 重建仿真环境 → RL 训练 → 部署
  优势：
   1. 模型可以"试错"，学到模仿学习学不到的策略
   2. 仿真可以扩增（变天气、变车流密度、变激进程度）
   3. 不再受限于"人类怎么开"
```

**所以数据闭环的"终极形态"**：

```
真实数据采集 
    ↓
3DGS 重建仿真环境
    ↓
模仿学习 + RL 后训练
    ↓
仿真闭环验证
    ↓
OTA 上车
    ↓
（继续采集真实数据）
```

## 9. 面试问题与答案

### Q1: 介绍一下你理解的数据闭环？

**标准答案（30秒）**：
> "数据闭环是端到端规划工程化的核心。它把车端实时遇到的数据（特别是接管、急刹、长尾场景）通过 trigger 筛选回传到云端，然后用大模型/离线算法自动生成训练所需的 GT（检测框、矢量地图、未来轨迹），重新训练后通过仿真验证再 OTA 推回车端。本质上就是把'训练数据准备'变成一个持续运转的系统。"

### Q2: 数据闭环和训练数据准备的关系？

**标准答案**：
> "训练数据准备是数据闭环的第一次循环——一次性采集 nuScenes 这种公开数据集训练 baseline。数据闭环是这个过程的持续运转版，依靠车队不断采集新数据。两者复用同一套数据格式（比如 VAD 的 `forward_pts_train` 那一长串输入），区别是闭环系统要解决三个新问题：
> 1. 海量数据中筛什么（trigger）；
> 2. 没人工标注怎么办（自动标注）；
> 3. 上车前怎么验证（仿真闭环）。"

### Q3: 实车数据如何重新生成训练所需的 GT？

**标准答案**：
> "分两类：
> - **可以事后计算的**：`ego_his_trajs` / `ego_fut_trajs` / `ego_fut_cmd` / `ego_lcf_feat`。这些都可以从车辆位姿日志和 CAN 数据切片得到——因为是事后回看，未来已经发生了。这是免费午餐。
> - **需要算法生成的**：`gt_bboxes_3d` 用离线大模型自动标注（参数量比训练模型大很多）；`map_gt_*` 用离线建图算法（多次经过同一路段融合）；`gt_attr_labels` 用离线追踪+后向预测。
> 
> 核心原理是'用大模型标小模型的数据'，这是各家车企的核心壁垒。"

### Q4: trigger 怎么设计？

**标准答案**：
> "trigger 设计要平衡两个目标：覆盖长尾 + 控制带宽。常见类型：
> 1. **强信号**：人类接管、AEB 触发、急刹/急转——这些一定要回传；
> 2. **不一致信号**：模型预测 vs 人类实际驾驶偏差大、影子模式下两个模型决策不同；
> 3. **不确定信号**：模型置信度低、约束 loss 突增；
> 4. **场景信号**：VLM 识别出施工/事故、规则系统告警；
> 5. **随机采样**：1/1000 兜底，保证常规场景也有覆盖。
> 
> 特斯拉号称有 221 个 trigger，覆盖各种 corner case。"

### Q5: 端到端规划的数据闭环和检测的有什么不同？

**标准答案**：
> "几个关键差异：
> 
> 1. **GT 没有唯一答案**：检测有标准 3D 框，但'人类驾驶的轨迹'只是一种合理选择，不是唯一最优。所以单纯模仿学习有上限。
> 
> 2. **接管事件的语义模糊**：人类接管可能是因为模型错了，也可能是人类紧张/有特殊需求。需要细粒度区分。
> 
> 3. **长尾场景的因果不易判断**：同样是急刹，原因可能是前车加塞 / 行人横穿 / 误检测。需要给数据打因果标签。
> 
> 4. **闭环验证更复杂**：检测可以离线算 mAP，规划必须仿真闭环（因为开环 L2 不能反映真实安全性）。
> 
> 这也是为什么后续工作（如 RAD）引入 RL + 3DGS 仿真。"

### Q6: 你怎么看 VAD 的部署落地？数据闭环要做什么？

**标准答案（结合代码深度）**：
> "VAD 论文代码的 `forward_test` 是评测用的，传入 GT 是为了算指标。要落地必须做几件事：
> 
> 1. **写部署 inference wrapper**：剥离评测代码，只接收传感器输入。`gt_*` 字段全部去掉，只保留 `img` / `img_metas` / `ego_his_trajs` / `ego_fut_cmd` / `ego_lcf_feat`。
> 
> 2. **接入数据闭环**：在推理流程中插 trigger 钩子，把值得回传的 clip 上传云端。
> 
> 3. **云端自动标注管线**：用大模型生成 `gt_bboxes_3d` / `map_gt_*` / `gt_attr_labels`，配合位姿日志切片生成 ego 相关字段，重新组装成 nuScenes 风格的 .pkl 文件。
> 
> 4. **仿真闭环验证**：每次 OTA 前在 3DGS 重建场景里跑全量回归测试。
> 
> 5. **持续迭代**：数据闭环让 VAD 不再是一次训练的静态模型，而是持续进化的系统。"

### Q7: 数据闭环的瓶颈在哪里？

**标准答案（展示工程视野）**：
> "几个常见瓶颈：
> 
> 1. **自动标注精度**：大模型再强也有错误率，错误的 GT 会污染训练。需要人工抽检 + 多模型 cross-check。
> 
> 2. **回传带宽**：单条 clip 几百 MB，6 相机原始数据更大。需要压缩 + 智能筛选。
> 
> 3. **存储成本**：百亿 clip 的存储和检索是工程难题。Tesla 自建 Dojo，小厂只能依赖云。
> 
> 4. **训练成本**：每次重训练要烧大量 GPU 时间。需要增量训练 / 持续学习方法。
> 
> 5. **仿真 sim-to-real gap**：仿真通过不代表实车安全，需要影子模式过渡。
> 
> 6. **数据治理**：用户隐私（人脸、车牌脱敏）、法规合规（地理围栏）。
> 
> 这些都是数据闭环工程师的实际工作内容。"

### 数据闭环的关键模块

```
车端：触发器（Trigger）
   ↓
云端：自动标注（Auto-labeling）
   ↓
云端：数据挖掘（Data Mining）
   ↓
云端：重训练 + 仿真验证
   ↓
OTA：模型部署回车端
```

### 数据闭环的"金句"

> **训练数据准备 = 数据闭环的第一次循环**
> 
> **部署推理产生数据 → 自动标注变回训练 GT → 重训练 → 再部署**
> 
> **核心原理：用大模型标小模型的数据**
> 
> **终极形态：真实数据 + 3DGS 仿真 + RL 后训练**
